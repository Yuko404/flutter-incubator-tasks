1. What is pub? What it does? Why do we need it?
Pub — это менеджер пакетов для Dart. Он нужен для того, чтобы удобно устанавливать любые пакеты, созданные сообществом. Разработчик пакеты загружает его на pub.dev,
чтобы любой другой разработчик смог найти пакет, изучить его и его документацию, загрузить и использовать.

2. What is pub spec? Which purpose does it serve?
Pub spec — файл конфигурации, который помогает настраивать зависимости и технические требования проекта. В нем устанавливается имя проекта, версия, какие пакеты используются и каких версий,
какая версия SDK требуется и т.п.

3. What is the purpose of pubspec.lock file? When and why it should be stored in [VCS], and when not? 
Цель файла pubspec.lock — «заблокировать» правильные версии пакетов, при которых запускается проект. Это критически важно, чтобы проект можно было б запустить на другом устройстве.
Из всех доступных версий выбирается одна, при которой нет проблем с зависимостями. При обновлении через dart pub upgrade или downgrade обновится и файл pubspec.lock. С недавнего
времени pubspec.lock хранит и hash пакетов, чтобы подменить их было невозможно.
Хранить данный файл в репозиториях типа github или других стоит, если вы создаёте проект или приложение, чтобы можно было запустить проект в другом месте. Лишним же этот
файл является, когда речь идет о создании пакета.

4. What does "version range" mean? How is it useful for dependencies?
Диапазон версий (или же ^) значит, что подойдет любая версия пакета из диапазона. Например, ^3.1.5 — >=3.1.5 <4.0.0. Нужно это для того, чтобы разрешать конфликты версий
зависимостей (какой-то пакет может работать только при определенной версии пакета, а другому может быть не так важно, какой версии пакет, итого система сама подберет подходящую
версию из диапазона).

5. Where is pub able to get dependencies from?
Pub может получать пакеты с таких мест, как pub.dev, git-репозитории, и даже локальные хранилища, SDK.

6. What is the difference between development dependencies and regular one? Which ones should be used and when?
Разница в том, что будет с пакетами при запуска приложения в прод. Dev packages будут работать только в то время, когда будет происходить разработка приложения. Во время самой
работы и запуска приложения в прод dev packages не будут использоваться и будут исключены. Поэтому нужно аккуратно использовать dev пакеты в проекте (например, в тестах или
примерах), так как они будут исключены. 

7. How [Dart] projects are structured in files? Which are common conventions and what for?
Структура проекта Dart следующая:
- pubspec.yaml - основная техническая информация проекта.
- lib/ - хранение исходного кода приложения.
- bin/ - хранение исполняемых файлов (точка входа в проект).
- test/ - хранение тестов приложения (unit-тесты, полный тесты, тесты виджетов и т.п.).
- assets/ - хранение изображений, шрифтов и т.п.
- README.md - обзор и описание проекта.

8. What do we need [Effective Dart] for? Why is it vital?
Effective Dart — золотые правила написание кода, которые считаются хорошим тоном среди разработчиков. Соблюдение данных правил не только помогает другим разработчикам, и вам через
время, лучше понимать ваш код, но и помогает избежать неточностей и ошибок во время его написания.

9. What is dart format used for? What are the benefits of using it?
dart format - инструмент для форматирования кода, который помогает переводить код в читаемый и понимаемый другими разработчиками код. Он помогает быстрее понимать код вам и другим
членам команды, позволяя экономить время по сравнению с разбором сути неотформатированного кода.

10. How commas are used to guide code formatting in [Dart]?
Запятые используются в форматировании Dart, чтобы обеспечить читаемость кода. Trailing commas — запятые, например, в списках, которые располагают элементы или вертикально, или
горизонтально (в зависимости от их наличия). Наличие Trailing commas в конце списка параметров заставляет dart format форматировать список вертикально (каждый параметр с новой строки), 
игнорируя длину строки. Отсутствие запятой позволяет форматтеру пытаться уместить всё в одну строку.

11. What is static analysis? What is linting? How they are represented in [Dart]? Why should we use them?
Статический анализ — анализ кода без его запуска (поиск ошибок, предупреждений и т.п.). В Dart для этого используется analyzer, в котором устанавливаются правила, при несоблюдении
которых, он сообщит об этом. В него также входит linter, в котором устанавливаются правила для стиля кода и наилучших практик.
Linting — это как раз процесс, который выполняет linter (то есть проверка кода на стилистические ошибки, баги, установленные правила хороших практик). Использовать 
статистический анализ критически важно не только для того, чтобы привести код к хорошему и понятному виду, но и также ускорить процесс написания кода путём отлова ошибок даже 
без запуска приложения.

12. Why source code documentation matters? How is it represented in [Dart]?
Документация кода — очень важная составляющая программирования. Важная она потому, что позволяет сразу объяснять, что это за функция и зачем она нужна. Это помогает не только 
разработчикам понять, как работает функция (что критически важно для любого сообщества), но и самому разработчику через время понять, о чём и зачем та или иная функция или класс. 
В dart она представлена комментариями ///, а также инструментом dart doc, который генерирует для пакета html документацию.

13. Which are good practices for documenting code and [API]s in [Dart]? 
Хорошие практики документации в Dart:
1. Использовать комментарии документации (///), а не обычные комментарии (//) или блоки комментариев (/*  */).
2. Докуменировать library директиву, публичные классы, методы, переменные и т.п. (публичный API).
3. Структурировать документацию (сначала 1 короткое приложение по-сути, далее пробельная строка и детальное описание, после ещё одой пробельной строки пример (опционально)).
4. Для функций с типом возвращаемого значения bool начинать документацию со слова Whether (Является ли), переменные с bool - с существительного.
5. Использовать [] для ссылки на переменные, классы, методы в документации.
6. Избегать тавтологий и излишних фраз.
7. Не документировать сеттер и геттер одновременно (только один из).

14. How can one publish and serve [API] documentation in [Dart]?
1. Сгенерировать локально через dart doc (+ открыть локально, например, через dhttpd), при изменении перегенерировать через dart doc.
2. Опубликовать на pub.dev (документация снегерируется автоматически при dart pub publish).
3. Опубликовать с помощью хостингов (например, Firebase, Github actions, etc).
